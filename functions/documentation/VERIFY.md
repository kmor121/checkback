# VERIFY.md - 回帰テスト手順書

## 運用ルール

-   **修正後の必須プロセス:** 機能追加やバグ修正を行った際は、必ずこの手順書に記載された関連テストを実施し、結果を記録すること。
-   **Verifyゲート:** 全ての関連テストが `OK` となるまで、該当のタスクや修正を `Done` としてはならない。`NG` が発生した場合は、開発者に差し戻す。
-   **記録の徹底:** テスト実施後は、必ず末尾の「実行記録」セクションに結果を追記する。

---

## 必須テスト項目 (最低5点)

### V-01: 既存コメント選択と描画表示

-   **目的:**
    -   ユーザーがコメントを選択した際に、UIのハイライトと `ViewerCanvas` 上の描画が正しく連動することを確認する。
-   **手順:**
    1.  複数の描画付きコメントが存在する `ShareView` を開く。
    2.  右パネルのコメントAをクリックする。
    3.  次に、右パネルのコメントBをクリックする。
    4.  再度、コメントAをクリックする。
-   **期待結果:**
    -   手順2: コメントAがハイライトされ、CanvasにコメントAの描画のみが表示される。
    -   手順3: コメントBがハイライトされ、CanvasにコメントBの描画のみが表示される（コメントAの描画は消える）。
    -   手順4: コメントAがハイライトされ、CanvasにコメントAの描画のみが表示される。
-   **観測ポイント:**
    -   `ShareView` の `activeCommentId` stateが、クリックしたコメントのIDに正しく更新されるか。
    -   `ViewerCanvas` に渡される `existingShapes` propが、選択されたコメントの描画のみを含んでいるか。
    -   コンソールログ `[ViewerCanvas] renderedShapes` の件数が期待通りか。
-   **NG時の切り分け:**
    -   ハイライトは変わるが描画が変わらない場合: `ShareView` から `ViewerCanvas` へのprops伝達の問題。
    -   描画が混在・残存する場合: `ViewerCanvas` 内部の `shapesMapRef` クリアロジック、または `renderedShapes` の計算ロジックの問題。

---

### V-02: 新規コメント入力時の選択抑制

-   **目的:**
    -   ユーザーが新規コメントのテキスト入力を開始した際に、他のUI要素の選択が確実に抑制されることを確認する。
-   **手順:**
    1.  複数のコメントが存在する `ShareView` を開く。
    2.  下部Composerの「コメントを入力...」(`Textarea`) をクリックする。
    3.  右パネルに表示されている任意のコメントカードをクリックする。
    4.  Composerの外側のエリアをクリックして、フォーカスを外す。
-   **期待結果:**
    -   手順2: 右パネルのどのコメントもハイライトされていない状態になる。
    -   手順3: コメントカードをクリックしても何も反応せず、ハイライトも付与されない。新規コメント入力モードが維持される。
-   **観測ポイント:**
    -   `ShareView` の `isNewCommentInputActive` stateが `true` になるか。
    -   `activeCommentId` が `null` に設定され、維持されるか。
    -   コンソールログ `[P0] selection suppressed` が表示されるか。
    -   `Textarea` の `onPointerDownCapture` で `stopPropagation` が呼ばれているか。
-   **NG時の切り分け:**
    -   コメントが選択されてしまう場合: `enterNewTextOnlyComposer` の実装、特に `stopPropagation` が機能していない可能性。または、イベントの捕捉タイミングの問題。

---

### V-03: ペイントモードON/OFF時の安定性

-   **目的:**
    -   ペイントモードを切り替えても、`ViewerCanvas` のズームやパンの状態がリセットされず、ちらつきが発生しないことを確認する。
-   **手順:**
    1.  `ShareView` を開き、ファイルを200%以上にズームする。
    2.  ドラッグで画像をパン（スクロール）し、表示位置を中央からずらす。
    3.  ツールバーの「ペイント」ボタンをONにする。
    4.  再度「ペイント」ボタンをOFFにする。
    5.  手順3-4を5回以上、高速で繰り返す。
-   **期待結果:**
    -   手順3-5を通じて、ズーム倍率とパンの位置が完全に維持される。
    -   画面が一瞬リセットされたり、ちらついたりしない。
-   **観測ポイント:**
    -   `ViewerCanvas` コンポーネント自体が再マウント（Unmount→Mount）されていないか（React DevToolsで確認）。
    -   `ViewerCanvas` の `zoom` および `pan` stateが、`paintMode` の変化によってリセットされていないか。
    -   コンソールログ `[ViewerCanvas] fileIdentity/pageNumber changed` が、ペイントモード切替時に**表示されない**こと。
-   **NG時の切り分け:**
    -   ズーム/パンがリセットされる場合: `ViewerCanvas` の `useEffect` の依存配列に `paintMode` が含まれている、または `Stage` コンポーネントが `key` 変更で再マウントされている可能性。

---

### V-04: 描画下書きの保持（描画 ↔ テキスト入力）

-   **目的:**
    -   描画モードとテキスト入力モードを行き来しても、作成中の描画下書きが失われないことを確認する。
-   **手順:**
    1.  `ShareView` で新規コメント作成を開始する。
    2.  ペイントモードをONにし、Canvas上にペンで何かを描画する。
    3.  下部Composerの「コメントを入力...」(`Textarea`) をクリックし、テキストを入力する。
    4.  再度、ツールバーの「ペイント」ボタンをONにし、ペイントモードに戻る。
-   **期待結果:**
    -   手順3: `Textarea` をクリックした際、手順2で描いた描画はCanvas上に表示されたままである。
    -   手順4: ペイントモードに戻った際も、手順2で描いた描画がCanvas上に保持されている。
-   **観測ポイント:**
    -   `ShareView` の `draftShapes` stateが、モード切替後も描画データを保持しているか。
    -   `localStorage` の `draftPaint:*` キーにデータが正しく保存されているか。
    -   `hidePaintOverlay` のロジックが、描画データが存在する場合には `false` を返し、レイヤーを非表示にしていないか。
-   **NG時の切り分け:**
    -   描画が消える場合: `draftShapes` stateが意図せずクリアされている、または `hidePaintOverlay` の条件分岐が不適切で、描画レイヤーが非表示にされている可能性。

---

### V-05: コメント送信処理の安定性 (handoff/freeze)

-   **目的:**
    -   描画付きの新規コメントを送信する際、`temp ID` から `real ID` への切り替え（handoff）と、DB反映までのUI保護（freeze）が機能し、描画のちらつきや消失が発生しないことを確認する。
-   **手順:**
    1.  ブラウザの開発者ツールで、ネットワークを「Slow 3G」などに設定する。
    2.  `ShareView` で新規コメントを作成し、複数の描画を行う。
    3.  テキストを入力し、「送信」ボタンをクリックする。
    4.  送信処理中（ネットワークが遅いため数秒かかる）の画面表示を注視する。
-   **期待結果:**
    -   送信ボタンクリック後、DBにデータが反映されるまでの間、Canvas上の描画は消えずに表示され続ける。
    -   送信完了後、新しいコメントが選択状態となり、描画も正しく表示される。
-   **観測ポイント:**
    -   送信中のコンソールログで `[P0-FLICKER] handoff` や `[P0-FIX] Freeze activated` が表示されるか。
    -   `ViewerCanvas` に渡される `shapesForCanvas` が、送信中に空の配列になっていないか。
    -   `handoffRef` と `freezeRef` が送信処理中に適切なデータを保持し、DB反映後に `null` にクリアされているか。
-   **NG時の切り分け:**
    -   描画がちらつく/消える場合: `handoffRef` または `freezeRef` の生成・解除タイミング、または `react-query` の `invalidateQueries` との連携に問題がある可能性。

## 追加推奨テスト項目

-   **V-A01: ページ・ズーム操作時の座標保持**
    -   目的: ページ切替やズーム/パン操作中に、コメントピンや描画注釈の座標がずれないことを確認。
    -   手順: (1) 複数ページのPDFで、各ページにコメントを配置。(2) ページを切り替え、ズームイン/アウト、パン操作を繰り返す。(3) 各コメントピンが正しい位置に留まるか確認。
-   **V-A02: 共有リンク権限の強制力**
    -   目的: `ShareLink` で設定された権限（閲覧のみ、コメント不可、DL不可）が、ゲストユーザーに対して正しく強制されることを確認。
    -   手順: (1) 権限を絞った共有リンクを3種類作成。(2) それぞれのリンクでゲストとしてアクセス。(3) コメント投稿ボタンやダウンロードボタンが非表示/無効になっているか、API直叩きが失敗するかを確認。
-   **V-A03: コメントフィルター機能**
    -   目的: 「未対応」「対応済」「すべて」のフィルターが、コメント一覧の表示を正しく切り替えることを確認。
    -   手順: (1) 未対応と対応済みのコメントが混在するファイルを開く。(2) 各フィルターをクリック。(3) 右パネルの一覧が正しくフィルタリングされるか確認。
-   **V-A04: 大量データ時の性能**
    -   目的: コメント数が100件以上、描画オブジェクトが500個以上の場合でも、UIの応答性が許容範囲内であることを確認。
    -   手順: (1) 大量データを持つテストファイルを用意。(2) コメント選択、スクロール、ペイントモード切替などの基本操作を行う。(3) 描画のカクつきやUIのフリーズが発生しないか体感で確認。
-   **V-A05: 例外系ハンドリング**
    -   目的: オフライン状態や権限不足など、異常系シナリオでアプリがクラッシュせず、適切なエラーメッセージを表示することを確認。
    -   手順: (1) 開発者ツールでネットワークをオフラインにし、コメントを投稿。(2) 権限のないユーザーでコメント編集を試みる。(3) それぞれ「送信失敗」や「権限がありません」といったトースト通知が表示されるか確認。
