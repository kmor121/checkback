# 開発・PR作成フロー

このドキュメントでは、PR作成からマージまでの標準フローを説明します。

---

## ⚠️ P0（回帰すると致命傷）領域の定義

以下の項目は、回帰するとユーザーの作業が失われたり、アプリケーションの根幹機能が利用不能になります。変更時は特に検証が必要です。詳細は [components/TEST_CHECKLIST.md](./TEST_CHECKLIST.md) を参照。

*   **A. Canvas ちらつき対策**
    *   A-1. 初回表示でのちらつき/白フレームの再発
    *   A-2. コメント切替時のちらつき/画像再ロード
    *   A-3. pending 中の shape の消失

*   **B. 新規ペイントのリサイズ保持**
    *   B-1. リサイズ直後の巻き戻り
    *   B-2. コメント切替後のリサイズ状態の消失
    *   B-3. 送信後のリサイズ情報の破損

*   **C. 編集モードで既存(DB)shape 選択→Delete 可能**
    *   C-1. 既存 shape の削除機能の不動作
    *   C-2. 下書き shape の削除機能の不動作

*   **D. 下書き仕様：×は保持、破棄は削除（new/edit 共通）**
    *   D-1. ×ボタンで下書きが消失する
    *   D-2. 破棄ボタンで下書きが削除されない
    *   D-3. 破棄ボタン操作後、削除対象の下書きが復活する

*   **F. リグレッションチェック（全体確認）**
    *   F-1. 新規→編集→新規の切り替え時に下書きが消失する
    *   F-2. 複数コメントの下書きが混同または消失する

## 📋 PR作成フロー（標準手順）

### 1. ブランチ作成・実装
```bash
git checkout -b feature/your-feature-name
# 実装...
git add .
git commit -m "feat: your feature description"
git push origin feature/your-feature-name
```

### 2. PR作成
1. GitHub上でPRを作成
2. **[components/PULL_REQUEST_TEMPLATE.md](./PULL_REQUEST_TEMPLATE.md) の内容をPR本文にコピー**
3. テンプレートに従って記入

### 3. スモークテスト実施（必須）
**⚠️ コードレビュー依頼前に必ず実施**

1. **[components/TEST_CHECKLIST.md](./TEST_CHECKLIST.md) を開く**
2. **該当するテスト項目を1周実施**
   - ShareView関連の変更 → セクション A〜F を全て実施
   - その他の変更 → 影響範囲に応じて該当項目を実施
3. **失敗した項目があれば PR本文に記載**
   - 再現手順、期待結果、実際の結果を明記
   - 可能であれば原因と対応方針も記載

### 4. レビュー依頼
- [ ] スモークテスト完了
- [ ] PR本文にテスト結果を記載済み
- [ ] Console に CRITICAL ERROR が出ていない

### 5. マージ
- レビュー承認後、マージ

## 🚀 PRの粒度と運用ガイドライン

### 1PRの適切な粒度

*   **原則**: 1つのPRは1つのテーマ（機能追加、バグ修正、UI改善など）に限定してください。
*   **機能追加**: 大きな機能は、小さなサブ機能に分割してPRを分割することを検討してください。
*   **回帰リスク**: 回帰リスクの高い変更（例: `ViewerCanvas.js` や `ShareView.js` のコアロジック変更）は、特に影響範囲を限定し、可能であれば影響の少ない部分からPRを分割してください。

### PR必須手順：テストチェックリストの運用

コードレビュー依頼前に、以下の手順を必ず実施してください。

1.  **PR本文に `components/PULL_REQUEST_TEMPLATE.md` の内容をコピー＆ペーストする。**
2.  **コピーしたPRテンプレートの「スモークテスト実施」セクションを参照し、`components/TEST_CHECKLIST.md` を開く。**
3.  **変更内容に応じて、`TEST_CHECKLIST` の該当項目をすべて実施する。**
    *   ShareView の描画やコメント機能に関する変更の場合は、セクション A〜F を全て実施してください。
    *   その他の変更の場合は、影響範囲を考慮し、関連する項目を実施してください。
4.  **PR本文の「スモークテスト結果」セクションに、実施結果を記載する。**
    *   全ての項目が成功した場合は、「全ての項目が✅です。」と記載。
    *   失敗した項目がある場合は、その項目番号、結果（✅成功/❌失敗）、再現手順、期待挙動、実際の結果、そして可能であれば考察や対応方針を明記してください。

### PR本文の最低記載項目

PR本文は、レビュアーが内容を素早く理解し、適切にレビューを行うために重要です。以下の項目を必ず含めてください。

*   **目的**: このPRが解決する課題、または追加する機能の目的。
*   **変更内容**: コードレベルでの主要な変更点（例: 〇〇ファイルを修正、△△関数を追加）。
*   **スモークテスト結果**: 上記「PR必須手順：テストチェックリストの運用」で実施した結果。
*   **影響範囲**: どのファイル、コンポーネント、または機能に影響があるか。
*   **レビュアーへの補足**: 特に見てほしいポイントや、懸念事項など。

## 📝 回帰防止の重要項目

以下の項目は過去にバグが発生した箇所です。**変更時は特に注意してください**：

### ShareView（ペイント機能）
- **Canvas ちらつき**: コメント切替時、画像の再ロードや白フレームが出ないこと
- **リサイズ保持**: 新規描画した図形をリサイズ後、元のサイズに戻らないこと
- **下書き保持**: ×ボタンで閉じた後、再度開いた時に下書きが復元されること
- **既存shape削除**: 編集モードで既存（DB保存済み）のshapeを削除できること

### ViewerCanvas（描画エンジン）
- **Map方式の不変更新**: `shapesMapRef.current` は必ず新しいMapインスタンスで置き換えること
- **comment_id の厳格管理**: `renderTargetCommentId` を最優先し、fallback禁止
- **描画中の座標固定**: `drawViewRef.current` で描画開始時のview/scaleを保存し、描画中は固定すること

## 🔧 開発時のTips

### DEBUG モードの使い方
URL に `?debug=1` を追加すると、詳細ログが表示されます：
```
https://your-app.base44.com/share?token=xxx&debug=1
```

### Console ログの見方
- `[ViewerCanvas]`: Canvas内部の状態
- `[ShareView]`: ShareView全体の状態
- `[DRAW_DEBUG]`: 描画処理の詳細
- `CRITICAL ERROR`: 即座に修正が必要なエラー
- `WARNING`: 警告（許容される場合もある）

---

## ❌ 禁止事項

- **推測修正**: 必ずコードを確認してから修正すること
- **関係ない修正**: バグ修正に無関係なリファクタ・整形・命名変更は禁止
- **テストスキップ**: スモークテスト未実施でのPR作成は禁止
